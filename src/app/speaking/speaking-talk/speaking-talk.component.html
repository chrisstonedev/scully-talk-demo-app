What is technical debt?

Technical debt refers to the parts of our code that is outdated and imperfect, often due to having taken shortcuts. Teams often tell themselves that they will go back and adjust things when other priority work is complete. When this is not done, this affects the quality of the application significantly and becomes harder to resolve, just as a debt can grow and become harder to pay off.

Types of technical debt include:

  • Taking a lot of shortcuts
  • Team faces too many last-minute changes
  • Data, UI, and business logic is all combined
  • Code is hard to read
  • Code is scary and making changes is too risky
  • There is no documentation
  • There is no cross-training on the team
  • There is no clear leadership or ownership
  • There are no unit tests
  • Concerns from developers are not taken into account
  • Dependencies are outdated
